# Product

## Register

product

## Users

stash is for a single developer or AI-tool power user working locally across a terminal, a todo list, and Claude or Codex sessions. They use it repeatedly throughout the day to capture work quickly, decide what to do next, carry project context into an agent session, and review what happened without reconstructing the story from separate tools.

## Product Purpose

stash connects todos, projects, agent sessions, decisions, lessons, and usage in one local-first workflow. Success means a user can move through capture, triage, execution, session evidence, and review without learning internal implementation categories or choosing between overlapping dashboards.

## Brand Personality

Focused, technical, direct. The interface should feel like a trusted local instrument: fast enough for keyboard-heavy use, expressive enough to retain the existing stash identity, and calm enough that the work remains more important than the chrome.

## Anti-references

- A design-demo gallery or switcher presented as product navigation.
- Dashboards that expose multiple overlapping visualizations as equal destinations.
- Navigation labels based on implementation names, letters, or internal route structure.
- Actions such as creating a project or starting a session disguised as permanent destinations.
- Context-free links to todo, project, or session detail pages.

## Design Principles

1. Organize around the user's workflow: work, projects, sessions, review, and settings.
2. Navigation answers where to go; buttons answer what to do; entity links preserve context.
3. Keep one canonical surface for each job and treat alternate representations as local view options, not separate products.
4. Make the next useful action obvious while preserving keyboard speed for experienced users.
5. Use real persisted data and explicit loading, empty, and error states; never imply unsupported behavior.

## Accessibility & Inclusion

Target WCAG AA for contrast, keyboard access, focus visibility, semantic landmarks, and accessible names. Preserve reduced-motion preferences, do not rely on color alone for state, and keep primary workflows usable at narrow viewport widths even though the product remains desktop-first.
